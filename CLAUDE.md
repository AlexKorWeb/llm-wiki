# LLM Wiki — Управляющий файл

> **Это шаблон.** Полная инструкция по установке и запуску — в [README.md](README.md) (EN) / [README.ru.md](README.ru.md) (RU).
> Язык вики настраивается строкой ниже — поменяй под себя.

Локальная LLM-вики для технических заметок (инструменты, паттерны, концепты, решения).
Метод: Karpathy-style knowledge base с операциями **INGEST / QUERY / LINT**.

Язык общения: русский. Код и технические термины — на английском.

## Структура

```
.
├── raw/                  # Исходные заметки (пользователь кладёт, LLM только читает)
│   ├── _inbox/           # Сюда Telegram-бот складывает пересланные посты
│   └── _assets/          # Картинки из обработанных постов
├── knowledge/
│   ├── concepts/         # Страницы понятий (что это, зачем, как работает)
│   ├── tools/            # Утилиты, библиотеки, плагины, сервисы, платформы (НЕ Skills)
│   ├── skills/           # Claude Code Skills — пакеты SKILL.md в ~/.claude/skills/
│   └── connections/      # Связи между темами (X vs Y, X + Y, паттерны)
├── daily/                # Дневные логи сессий (автоматические записи)
├── scripts/              # Telegram-бот и служебные скрипты (tg_bot.py, flush.py, compile.py)
├── index.md              # Навигационный индекс (обновляется при каждом INGEST)
├── log.md                # История операций (append-only)
└── CLAUDE.md             # Этот файл — схема и инструкции
```

## Конвенции именования

- Файлы: **kebab-case** с расширением `.md` — например: `claude-code-hooks.md`, `python-pathlib.md`
- Без пробелов, без кириллицы в именах файлов
- Папки: только латиница, lowercase

## Формат статей — YAML Frontmatter

Каждая статья в `knowledge/` начинается с:

```yaml
---
title: "Название статьи"
tags: [tag1, tag2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: []          # ссылки на raw/-файлы или URL
related: []          # ссылки на другие knowledge/-страницы
---
```

После frontmatter — содержание в Markdown.

## Главное правило для всех INGEST

**raw/-файлы должны быть ПОДРОБНЫМИ** — это полноценные конспекты с URL-источниками, командами установки, примерами использования, сравнениями. Не 2-строчные резюме.

**knowledge/-страницы могут быть короче**, но **ОБЯЗАНЫ** ссылаться на URL-источники сразу после заголовка.

**НИКОГДА не теряй ссылки** на GitHub, официальные сайты, документацию. Если в исходном посте нет ссылок (например, Telegram их вырезал) — найди их через WebSearch и WebFetch.

## Операция INGEST из `raw/_inbox/` (автоматический поток)

Telegram-бот (`scripts/tg_bot.py`) принимает пересланные посты и кладёт их в `raw/_inbox/<timestamp>_<источник>.md`. Картинки — в `raw/_inbox/assets/<slug>/`. Бот вызывает `claude -p` ОТДЕЛЬНО для каждого файла.

При обработке inbox-файла:

1. **Прочитай файл**, определи тему и упомянутые сущности (инструменты, репо, сайты)
2. **Обогати через WebSearch/WebFetch** — найди канонические URL для всего упомянутого
3. **Реши: новая или дополнение?** — сравни с `raw/` через `index.md`
4. **Создай ПОДРОБНЫЙ raw/-файл** или дополни существующий (структура: Источники, Что это, Установка, Использование, Сравнение, Применение)
5. **Перенеси картинки** из `_inbox/assets/` в `raw/_assets/<тема>/`, обнови пути
6. **Создай knowledge/-страницу** с URL-источниками сразу после заголовка
7. **Обнови related** в 2-3 смежных страницах
8. **Обнови `index.md`** и **дополни `log.md`**
9. **Удали обработанные файлы** из `raw/_inbox/` (это делает бот после проверки)

## Операция INGEST (ручная)

Когда пользователь кладёт новый файл в `raw/` и просит обработать:

1. **Читай источник** — прочитай файл из `raw/` целиком
2. **Пиши summary** — создай страницу в `knowledge/` по смыслу:
   - `concepts/` — понятия, методики, техники
   - `tools/` — утилиты, библиотеки, плагины, сервисы, CLI, платформы
   - `skills/` — **только** Claude Code Skills (пакеты SKILL.md в `~/.claude/skills/`)
   - `connections/` — связи между темами (X vs Y, X + Y, паттерны)
   - Используй YAML frontmatter
   - В `sources:` укажи путь к raw/-файлу
   - Пиши кратко, структурированно, с примерами кода где уместно
3. **Обновляй связи** — найди связанные страницы (до 10–15 файлов за раз), добавь перекрёстные ссылки в `related:`
4. **Обновляй index.md** — добавь новую статью в соответствующий раздел
5. **Пиши в log.md** — append-запись формата:
   ```
   ## [YYYY-MM-DD] ingest | Название статьи
   - Источник: raw/filename.md
   - Создано: knowledge/category/page-name.md
   - Обновлены связи: [список файлов]
   ```

## Операция QUERY

Когда пользователь задаёт вопрос:

1. Прочитай `index.md` для навигации
2. Найди релевантные страницы в `knowledge/`
3. Ответь с цитатами и ссылками на конкретные страницы
4. **Если ответ содержит новое знание** — сохрани как новую страницу в `knowledge/` и обнови index

## Операция LINT

Проверка здоровья вики:

1. **Битые ссылки** — все `related:` и `sources:` указывают на существующие файлы
2. **Орфаны** — страницы в `knowledge/`, которые не упомянуты в `index.md`
3. **Противоречия** — конфликтующая информация между страницами
4. **Устаревшие данные** — страницы, не обновлявшиеся более 30 дней
5. Результат: отчёт с рекомендациями, что исправить

## Правила

- Не удаляй файлы из `raw/` — это источники истины пользователя
- Не редактируй файлы в `raw/` — только читай
- `log.md` — записи идут от новых к старым (новая запись вставляется СВЕРХУ, сразу после заголовка `# Wiki Log` и подсказки). Никогда не удаляй существующие записи.
- При конфликте между `raw/` и `knowledge/` — `raw/` приоритетнее
- Хорошие ответы на вопросы (QUERY) тоже сохраняются как страницы
- Максимум 10–15 файлов за одну операцию INGEST
